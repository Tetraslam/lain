# lain — plan

graph-based ideation engine. start with a seed idea, branch into n children, recurse to depth m. agents expand each node. named after lain iwakura.

## core concept

```
seed idea (root node)
├── branch 1 (agent explores direction A)
│   ├── branch 1.1 ──crosslink──┐
│   ├── branch 1.2               │
│   └── branch 1.3               │
├── branch 2 (agent explores direction B)
│   ├── branch 2.1 <─────────────┘
│   └── ...
└── branch n
    └── ...depth m

→ final synthesis pass across full graph
```

the graph is a **DAG** (directed acyclic graph), not a tree. nodes have a primary parent (forming the tree backbone) but can have cross-links to nodes in other branches. a final synthesis pass can traverse the full graph to identify connections, contradictions, and emergent patterns.

parameters:
- **seed**: the root idea (text, file, or interactive prompt)
- **n**: branching factor (children per node)
- **m**: max depth
- **extensions**: pluggable modules that shape how nodes are generated (worldbuilding, research, debate, etc.)

each node is generated by an agent that receives its ancestor chain as context and produces a meaningful divergence from siblings.

### core engine (`packages/core`)

the heart of lain. framework-agnostic, no UI concerns.

- **graph model**: DAG with tree backbone. each node contains:
  - id (deterministic, based on position in tree)
  - content (markdown)
  - metadata (generation params, model used, timestamp, extension data)
  - parent ref (primary, forms the tree)
  - children refs
  - crosslinks (refs to nodes in other branches, bidirectional)
  - status (pending, generating, complete, pruned)
- **orchestrator**: manages the expansion loop
  - takes seed + n + m
  - spawns agents **breadth-first or depth-first** (configurable, `--strategy bf|df`)
  - supports concurrency control (max parallel agents)
  - emits events for UI consumption (node:created, node:generating, node:complete, etc.)
- **synthesis pass**: after generation completes (or on-demand), traverses the full graph to:
  - identify cross-links between branches (AI-suggested)
  - surface contradictions or tensions
  - generate a summary/synthesis node at the root level
  - produces **annotations** (suggested cross-links, contradiction notes, synthesis summary)
  - annotations are staged, not applied — user reviews and merges via `lain merge-synthesis`
  - `--auto-merge` flag (or config option) to apply annotations automatically
  - configurable: `lain synthesize <id> --auto-merge` or `lain config set synthesis.auto_merge true`
- **storage**: SQLite as canonical format (see [data format](#data-format) below)
- **obsidian sync**: bidirectional sync between SQLite db and obsidian markdown folder (see [sync protocol](#sync-protocol) below)
- **operations**: prune, graft (attach subtree), redirect (regenerate from a node), merge, diff between branches, link (add cross-link)

### agent layer (`packages/agents`)

- **provider abstraction**: uniform interface over:
  - **anthropic API** (native, via @anthropic-ai/sdk)
  - **amazon bedrock** (native, for claude models on bedrock — first-class support)
  - **openai API** (native + compatible endpoints: ollama, together, openrouter, groq, etc.)
  - configurable per-node or per-depth (e.g., opus via bedrock for root expansions, haiku for deeper nodes)
- **prompt engine**:
  - constructs prompts from ancestor chain + sibling context + extension directives
  - sibling-awareness: each agent knows what its siblings are exploring so it diverges meaningfully
  - **branching plan phase** with configurable detail level:
    - `brief`: ~5 words per direction (fastest, cheapest)
    - `sentence`: 1 sentence per direction
    - `detailed`: 2-3 sentences per direction (best diversity, most tokens)
    - `none`: skip planning, generate directly
  - configurable system prompts, temperature, and sampling params per depth level
- **concurrency**: agent pool with configurable parallelism
  - respect rate limits per provider
  - retry with backoff
  - streaming support for real-time UI updates

### extension system (`packages/extensions`)

extensions are maximally powerful plugins. they can modify nearly every aspect of lain's behavior.

**capabilities:**
- inject system prompt fragments (per-depth, per-node, global)
- add constraints or directives at specific depths
- define custom node types with additional fields and validation
- hook into **all** lifecycle events:
  - `before:plan`, `after:plan` — branching plan phase
  - `before:generate`, `after:generate` — node content generation
  - `before:prune`, `after:prune` — pruning
  - `before:link`, `after:link` — cross-linking
  - `before:synthesize`, `after:synthesize` — synthesis pass
  - `before:sync`, `after:sync` — obsidian sync
  - `before:export`, `after:export` — any export operation
  - `on:error` — error handling
- define **custom graph operations** (registered as CLI subcommands):
  - e.g., worldbuilding extension registers `lain worldbuilding timeline <exploration-id>` that traverses all history-tagged nodes and generates a chronological timeline
  - e.g., research extension registers `lain research bibliography <exploration-id>` that collects all citations across the graph
- define **custom synthesis strategies** (replace or augment the default synthesis pass)
- provide custom UI components (for web/TUI)
- define custom node **renderers** for obsidian export (control how their node types serialize to markdown)
- register custom **validators** that run before/after generation (e.g., "all worldbuilding nodes must have a `region` field")
- access the full graph API (read any node, query by depth/type/extension_data, traverse)
- spawn their own agent calls (extensions can call LLMs independently)

**extension API shape (sketch):**

```typescript
interface LainExtension {
  name: string;
  version: string;

  // credentials this extension needs (user provides via `lain extensions auth`)
  requiredAuth?: { key: string; description: string; required: boolean }[];

  // extension-specific config schema (presented via `lain extensions config`)
  configSchema?: ConfigFieldDefinition[];

  // prompt injection
  systemPrompt?: (context: NodeContext) => string;
  planPrompt?: (context: PlanContext) => string;

  // lifecycle hooks (all optional)
  hooks?: Partial<LifecycleHooks>;

  // custom node types
  nodeTypes?: NodeTypeDefinition[];

  // custom graph operations (become CLI subcommands)
  operations?: OperationDefinition[];

  // custom synthesis strategy
  synthesizer?: (graph: GraphAPI) => Promise<SynthesisResult>;

  // custom obsidian renderer
  renderer?: (node: Node, context: RenderContext) => string;

  // validators
  validators?: ValidatorDefinition[];

  // web/TUI UI components
  ui?: UIComponentDefinition[];
}
```

built-in extensions:
- **worldbuilding**: geography, cultures, history, magic systems. custom ops: `timeline`, `atlas`, `faction-map`. custom node types with region/era/faction fields.
- **research**: academic-style exploration with citations, methodology branching. custom ops: `bibliography`, `methodology-tree`. validators ensure citation format.
- **debate**: adversarial branching (pro/con/steelman/critique). custom synthesis that produces a "verdict" node weighing all arguments.
- **fiction**: character arcs, plot branching, narrative exploration. custom ops: `character-sheet`, `plot-summary`. custom node types with POV/scene fields.
- **engineering**: technical design exploration, tradeoff analysis. custom ops: `decision-matrix`, `adr-export` (architecture decision records).
- **freeform**: no constraints, pure divergent thinking (default)

**extension distribution & auth:**
- extensions are npm packages that export a `LainExtension` object as their default export
- installed via `lain extensions add <package-name>` (runs `pnpm add` under the hood into a managed extensions directory)
- local extensions supported via `lain extensions add ./path/to/folder`
- extensions can declare **required credentials** in their manifest (e.g., a research extension needs a Semantic Scholar API key, a web-scraping extension needs a proxy key)
- `lain extensions auth <extension> --set KEY=value` stores credentials in the lain config (at `~/.config/lain/extensions/<name>/auth.json`, encrypted at rest via OS keychain if available, plaintext fallback with a warning)
- extensions access their credentials via `context.auth.get("KEY")` — they never handle storage themselves
- `lain extensions config <extension>` opens an interactive clack prompt for extension-specific settings (defined by the extension via a `configSchema` field)

### CLI (`packages/cli`)

primary interface. built with **clack** for beautiful interactive prompts.

```bash
# start a new exploration
lain "what if we built cities underwater" --branches 5 --depth 3

# from a file
lain explore --seed ideas/underwater-cities.md --n 4 --m 6

# with extension and generation strategy
lain "an empire built on fungal networks" --ext worldbuilding --n 3 --m 4 --strategy df

# with branching plan detail level
lain "quantum computing applications" --n 5 --m 3 --plan detailed

# interactive mode (clack prompts guide you through setup)
lain interactive

# init (workspace + global config)
lain init                            # set up global config at ~/.config/lain/ (first run)
lain init --workspace                # create .lain/ in current dir with local config overrides

# tree management
lain status                      # show active explorations
lain show <node-id>              # display a node's content
lain tree <exploration-id>       # print tree structure
lain prune <node-id>             # mark node + descendants as pruned
lain extend <node-id> --n 3     # generate more children from a node
lain redirect <node-id>          # regenerate a node with new params
lain link <node-a> <node-b>     # add a cross-link between nodes
lain synthesize <exploration-id> # run synthesis pass across full graph
lain merge-synthesis <synth-id>  # apply synthesis annotations to graph
lain synthesize <id> --auto-merge # synthesize and apply in one step

# obsidian sync (bidirectional, first-class)
lain sync exploration.db              # bidirectional sync (default)
lain sync --push exploration.db       # db → markdown only
lain sync --pull exploration.db       # markdown → db only
lain sync --status exploration.db     # show what's changed since last sync
lain conflicts exploration.db         # list/resolve conflicts

# one-shot export (no sync tracking, just dump markdown)
lain export exploration.db            # export to folder alongside db
lain export exploration.db --out ~/vaults/tetrawiki/lain/  # custom output path

# config (global or local)
lain config set model claude-sonnet-4-20250514                          # global
lain config set provider bedrock --region us-east-1                     # global
lain config set provider openai-compatible --base-url http://localhost:11434
lain config set --local model claude-haiku-4-20250414                      # workspace-local override
lain config list                                                        # show effective config (merged)
lain config list --global                                               # show only global
lain config list --local                                                # show only local overrides

# extension management
lain extensions list                          # list installed extensions
lain extensions add @lain/worldbuilding       # install from npm registry
lain extensions add ./my-local-extension      # install from local folder
lain extensions remove @lain/worldbuilding
lain extensions auth <extension>              # configure auth/keys for an extension
lain extensions auth @lain/research --set SEMANTIC_SCHOLAR_KEY=sk-...
lain extensions config @lain/worldbuilding    # interactive extension-specific config

# daemon
lain watch exploration.db                     # auto-sync on file changes (foreground)
lain watch exploration.db --daemon            # background mode
lain watch --stop exploration.db
lain watch --status exploration.db
```

## config hierarchy

lain uses a three-tier config system, inspired by git. each tier overrides the one above it.

### 1. global config (`~/.config/lain/`)

created on first run or via `lain init`. applies to all explorations everywhere.

```
~/.config/lain/
├── config.json           # global settings (default model, provider, etc.)
├── credentials.json      # provider API keys (anthropic, bedrock, openai)
└── extensions/
    ├── @lain/
    │   └── worldbuilding/
    │       ├── auth.json     # extension credentials
    │       └── config.json   # extension settings
    └── ...
```

`config.json` example:
```json
{
  "default_model": "claude-sonnet-4-20250514",
  "default_provider": "bedrock",
  "bedrock": {
    "region": "us-east-1"
  },
  "default_n": 3,
  "default_m": 3,
  "default_strategy": "bf",
  "default_plan_detail": "sentence",
  "watch": {
    "debounce_ms": 500,
    "on_delete": "prune"
  },
  "synthesis": {
    "auto_merge": false
  }
}
```

### 2. workspace config (`.lain/`)

created via `lain init --workspace` in any directory. overrides global config for all explorations in that directory tree. useful when you have a vault or project with specific defaults.

```
~/vaults/tetrawiki/
├── .lain/
│   └── config.json       # workspace overrides (e.g., always use worldbuilding extension)
├── lain-explorations/
│   ├── zindabuul.db
│   └── zindabuul/
└── ...
```

workspace `config.json` example (sparse — only what you're overriding):
```json
{
  "default_ext": "worldbuilding",
  "default_model": "claude-sonnet-4-20250514",
  "default_n": 4,
  "default_m": 5
}
```

lain walks up the directory tree to find the nearest `.lain/` directory (like git finds `.git/`). if none is found, global config only.

### 3. exploration config (in the `.db` file)

each exploration stores its own config in the `exploration` table. these are set at creation time (from CLI flags) and override everything above. `lain config set --exploration <id> key value` can modify them after creation.

### resolution order

```
exploration db  >  workspace .lain/  >  global ~/.config/lain/  >  built-in defaults
```

`lain config list` shows the effective merged config and where each value comes from.

### `lain init`

```bash
lain init                    # interactive setup: walks you through provider config,
                             # API keys, default model, etc. writes to ~/.config/lain/
                             # idempotent — safe to re-run, only overwrites what you change

lain init --workspace        # creates .lain/ in current directory with workspace config
                             # clack prompts for what to override
```

on first run of any `lain` command, if `~/.config/lain/` doesn't exist, lain automatically runs the `init` flow (via clack prompts) so you're never stuck with a missing config.

## data format

### canonical storage: SQLite

each exploration is a single `.db` file. you put it wherever you want — inside an obsidian vault, in a project folder, wherever.

```
~/vaults/tetrawiki/lain-explorations/zindabuul-worldbuilding.db
~/projects/research/quantum-computing.db
~/Desktop/random-idea.db
```

SQLite because:
- single file (portable, backupable, git-friendly with LFS)
- fast random access to individual nodes (important during generation)
- supports indexing and queries ("find all depth-3 worldbuilding nodes")
- battle-tested TS support via `better-sqlite3`
- no server process needed
- way better than JSON for graphs with hundreds of nodes (no full-file rewrite on every update)

schema (simplified):

```sql
CREATE TABLE exploration (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL,
  seed TEXT NOT NULL,
  n INTEGER NOT NULL,
  m INTEGER NOT NULL,
  strategy TEXT DEFAULT 'bf',       -- bf or df
  plan_detail TEXT DEFAULT 'sentence',
  extension TEXT DEFAULT 'freeform',
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
);

CREATE TABLE node (
  id TEXT PRIMARY KEY,
  exploration_id TEXT NOT NULL REFERENCES exploration(id),
  parent_id TEXT REFERENCES node(id),
  content TEXT,                      -- markdown body
  content_conflict TEXT,             -- preserved db version when sync conflict occurs
  title TEXT,                        -- generated title
  depth INTEGER NOT NULL,
  branch_index INTEGER NOT NULL,
  status TEXT NOT NULL DEFAULT 'pending',  -- pending, generating, complete, pruned
  model TEXT,                        -- which model generated this
  provider TEXT,                     -- anthropic, bedrock, openai, etc.
  plan_summary TEXT,                 -- the branching plan summary for this direction
  extension_data TEXT,               -- JSON blob for extension-specific metadata
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
);

CREATE TABLE crosslink (
  source_id TEXT NOT NULL REFERENCES node(id),
  target_id TEXT NOT NULL REFERENCES node(id),
  label TEXT,                        -- optional description of the relationship
  ai_suggested BOOLEAN DEFAULT FALSE,
  created_at TEXT NOT NULL,
  PRIMARY KEY (source_id, target_id)
);

CREATE TABLE synthesis (
  id TEXT PRIMARY KEY,
  exploration_id TEXT NOT NULL REFERENCES exploration(id),
  content TEXT NOT NULL,             -- synthesis summary
  model TEXT,
  status TEXT NOT NULL DEFAULT 'pending',  -- pending, complete
  merged BOOLEAN DEFAULT FALSE,      -- whether annotations have been applied to graph
  created_at TEXT NOT NULL
);

CREATE TABLE synthesis_annotation (
  id TEXT PRIMARY KEY,
  synthesis_id TEXT NOT NULL REFERENCES synthesis(id),
  type TEXT NOT NULL,                -- 'crosslink', 'contradiction', 'note', 'merge_suggestion'
  source_node_id TEXT REFERENCES node(id),
  target_node_id TEXT REFERENCES node(id),  -- for crosslink type
  content TEXT,                      -- annotation body
  merged BOOLEAN DEFAULT FALSE,      -- whether this specific annotation has been applied
  created_at TEXT NOT NULL
);
```

### obsidian sync (bidirectional)

`lain sync <file.db>` is the primary command. it works like a simplified git:

**how it works:**

1. **on export** (`lain sync --push` or first `lain sync`): lain writes markdown files AND stores a **content hash** per node in the db (in a `sync_state` table). this is the "last known state" of each file.

2. **on import** (`lain sync --pull`): lain reads each markdown file, computes its hash, compares against the stored hash:
   - **hash matches**: file unchanged since last sync, skip
   - **hash differs**: file was edited externally (by you, by an agent, by obsidian). lain parses the frontmatter + content and updates the db.
   - **file missing**: node was deleted in obsidian. lain marks it as pruned in db (or deletes, configurable).
   - **new file**: a markdown file exists in the folder that doesn't correspond to any node. lain can import it as a new node (if it has valid frontmatter) or ignore it.

3. **on bidirectional sync** (`lain sync`, the default): lain does both — pulls external changes from markdown into db, then pushes any db changes back to markdown. conflict resolution:
   - **db changed + file unchanged**: db wins, overwrite file
   - **file changed + db unchanged**: file wins, update db
   - **both changed**: conflict. lain keeps both versions — the file version goes into the db as the current content, the db version is saved as `node.content_conflict` so nothing is lost. user resolves via `lain conflicts <exploration-id>`.

4. **sync state table** (added to schema). content and frontmatter are tracked separately so a metadata-only change (e.g., adding a tag in obsidian) doesn't conflict with a content edit from an agent:

```sql
CREATE TABLE sync_state (
  node_id TEXT PRIMARY KEY REFERENCES node(id),
  file_path TEXT NOT NULL,              -- relative path to the markdown file
  content_hash TEXT NOT NULL,           -- hash of markdown body at last sync
  frontmatter_hash TEXT NOT NULL,       -- hash of YAML frontmatter at last sync
  db_content_hash TEXT NOT NULL,        -- hash of db content at last sync
  db_frontmatter_hash TEXT NOT NULL,    -- hash of db frontmatter at last sync
  synced_at TEXT NOT NULL
);
```

this split means: if you edit frontmatter in obsidian (add a tag, change status) and an agent edits the content body, those are **not** a conflict — both changes merge cleanly. conflicts only occur when both sides touch the *same region* (both edited content, or both edited frontmatter).

**CLI:**
```bash
lain sync exploration.db              # bidirectional sync (default)
lain sync --push exploration.db       # db → markdown only
lain sync --pull exploration.db       # markdown → db only
lain sync --status exploration.db     # show what's changed since last sync
lain conflicts exploration.db         # list/resolve conflicts
```

**why this works for agents:**
the markdown files are just files on disk. any agent (claude, cursor, custom scripts, whatever) can read and edit them. next time you run `lain sync`, those edits flow back into the db. the agent doesn't need to know about lain at all — it just sees markdown files with frontmatter.

### file watcher daemon

`lain watch` runs a background daemon that auto-syncs on file changes. this is the live workflow: you have obsidian open, agents running, and lain keeps everything in sync automatically.

**how it works:**

1. uses `chokidar` (or Node's native `fs.watch` with a wrapper) to watch the markdown folder
2. **debouncing**: file changes are batched in a 500ms window. if obsidian autosaves or an agent edits 20 files in a burst, lain waits for the burst to settle before syncing. configurable via `lain config set watch.debounce_ms 1000`.
3. **self-write detection**: when lain itself writes files (during `--push`), it sets an internal "ignore" flag on those paths so the watcher doesn't trigger a feedback loop. implemented via a simple in-memory set of "paths I just wrote" that clears after the debounce window.
4. **lock file**: `<exploration>.db.lock` prevents concurrent writes. if `lain watch` is running and you manually run `lain sync`, the manual sync waits for the watcher's current batch to finish.
5. **notifications**: the daemon emits events that the TUI/web UI can subscribe to (via IPC or WebSocket) for real-time "file changed" indicators.

**daemon management:**
```bash
lain watch exploration.db             # start watching (foreground)
lain watch exploration.db --daemon    # start watching (background, writes PID file)
lain watch --stop exploration.db      # stop the background watcher
lain watch --status exploration.db    # is a watcher running? what's it seen?
```

**what it watches:**
- the markdown folder corresponding to the db
- only `.md` files (ignores images, .obsidian/, etc.)
- respects `.lainignore` if present (like .gitignore, for files in the folder that aren't lain nodes)

**when it syncs:**
- file created → `sync --pull` for that file (import new node)
- file modified → `sync --pull` for that file (update db)
- file deleted → mark node as pruned in db (configurable: `lain config set watch.on_delete prune|ignore|ask`)
- db changed (via `lain extend`, `lain redirect`, generation, etc.) → `sync --push` for affected nodes

the daemon is lightweight — it's just a file watcher + the existing sync logic. no separate server process, no ports. for the web UI, a thin API server runs separately and can talk to the watcher via the lock file and db.

`lain --obsidian <file.db>` reads the db and writes a folder of markdown files next to it (or at `--out`).

```
tetrawiki/lain-explorations/
├── zindabuul-worldbuilding.db           # canonical data
└── zindabuul-worldbuilding/             # synced obsidian folder
    ├── _index.md                        # exploration overview + synthesis
    ├── root.md
    ├── root-1.md
    ├── root-1-1.md
    ├── root-1-2.md ──wikilink──→ root-2-1.md  (crosslink)
    ├── root-2.md
    ├── root-2-1.md
    └── ...
```

the db has everything needed to regenerate the obsidian folder at any time. but edits to the markdown files are **first-class** — `lain sync` pulls them back in.

## open design questions

1. **branching diversity**: how do we ensure n children are meaningfully different?
   - **decided**: two-phase branching plan approach with configurable detail level (`brief`/`sentence`/`detailed`/`none`)
   - plan phase generates n direction summaries at chosen verbosity
   - expansion phase fills in each direction independently
   - user can review/edit plan before expansion begins (in interactive mode)

2. **context window management**: at depth m, the ancestor chain could be very long
   - summarize ancestors beyond a certain depth?
   - sliding window with full root + summarized middle + full parent?
   - let the user configure context strategy per extension?

3. **cross-linking**: ~~should nodes be able to reference nodes in other branches?~~
   - **decided**: yes, DAG with cross-links
   - AI-suggested during synthesis pass (staged as annotations, merged on user approval or auto-merge)
   - manually addable via `lain link`
   - rendered as wikilinks in obsidian export

4. **incremental generation**: should lain generate the full tree at once or let users expand interactively?
   - both: `lain "idea" -n 3 -m 4` generates the full tree, but `lain extend <node>` lets you grow it manually
   - interactive mode in TUI could let you approve/reject/redirect at each depth

5. **cost awareness**: deep trees with large n get expensive fast
   - n=5, m=4 = 5 + 25 + 125 + 625 = 780 nodes
   - need cost estimation before generation
   - tiered model strategy (expensive model for shallow, cheap for deep)

6. **collaboration**: multiple users expanding the same tree?
   - out of scope for v1, but SQLite file is easy to share/sync

7. **evaluation/ranking**: can lain help you identify the most promising branches?
   - separate "evaluator" agent that scores nodes?
   - user ratings that propagate?

8. **obsidian round-trip**: ~~should edits to exported markdown files be importable back into the db?~~
   - **decided**: yes, first-class bidirectional sync
   - `lain sync` handles push/pull/conflict detection
   - hash-based change tracking, no obsidian plugin required
   - agents and humans edit markdown freely, `lain sync` reconciles
